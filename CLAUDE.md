Project: product-monitor (single-file script, product-monitor.py)
Asyncio-based Python scraper that watches 4 Hungarian marketplaces (HardverApro, Moly, Vatera, Jófogás) for used-item listings matching configured search terms under a max price, and emails notifications about new listings/price changes. Sibling project of house-monitor (same author, same Gmail account, same Raspberry Pi).

Workflow: after any non-trivial modification to `product-monitor.py` (or its tests), run a full consistency check before calling the change done — `.venv/bin/python -m pytest tests/ -v` (all tests must still pass), plus `.venv/bin/python scripts/live_smoke_check.py` if the change touched a scraper's parsing/selectors (only that catches a live site redesign, see the Jofogas incident below). Treat any new failure as a regression to fix, not something to route around.

Dependency updates (Dependabot, `.github/dependabot.yml`): review every PR manually before merging, never auto-merge, even on a green CI run. A green CI run on a Dependabot PR only proves the network-free unit tests still pass — it does NOT prove the real scrapers still work, since a version bump of `aiohttp`/`beautifulsoup4`/`aiofiles` can change real network/parsing behavior that no unit test exercises. This isn't theoretical: during the step-4 quality pass (2026-07-03), an unrelated `pip install` had already silently resolved `aiohttp` to 3.14.1 - the exact version house-monitor documented as causing an intermittent `ClientPayloadError` regression under this same shared-`ClientSession` + `asyncio.gather`+`Semaphore` concurrency pattern. Rule going forward: any `aiohttp`/`beautifulsoup4`/other network-facing dependency PR needs a `scripts/live_smoke_check.py` run against the live sites (ideally 2+ times, since this failure mode is intermittent) before merging, not just a green CI check.

Production location:
- Script: `/opt/product-monitor/product-monitor.py` (systemd runs this file directly — no package/venv; `ExecStart=/usr/bin/python3 /opt/product-monitor/product-monitor.py`)
- Config: `/opt/product-monitor/product-monitor-config.json` (gitignored — real search terms are personal data; `product-monitor-config.example.json` is the committed placeholder)
- Log: `tail -f /var/log/product-monitor/service.log`
- Service: `sudo systemctl status/restart product-monitor`
- DB: `/opt/product-monitor/seen-products.json` (gitignored)
- Unit file: `/etc/systemd/system/product-monitor.service` — NOT reachable via the sshfs `/opt` mount or SSH from a Claude Code session on this machine; must be read/edited from the Pi's own terminal (`sudo systemctl cat product-monitor`)

Scrapers (4): HardverApro, Moly, Vatera, Jofogas — one class per site, each extends `BaseScraper`. All logic (dataclasses, scrapers, notifier, orchestrator) currently lives in the single `product-monitor.py` file (~950 lines) — not yet modularized like house-monitor's `house_monitor/` package. Modularization is planned but not started (see quality-pass status below).

Email: Gmail SMTP — `EMAIL_CONFIG` (module-level dict in `product-monitor.py`) reads from env vars (`PRODUCT_MONITOR_SMTP_SERVER/PORT/SENDER_EMAIL/SENDER_PASSWORD/RECIPIENT_EMAIL`), no hardcoded credentials in code (fixed 2026-07-03; previously these lived in plaintext inside `product-monitor-config.json`).
